üî∂ 1. üìå Derived Query (Method Name Query)

‚úÖ What is it?

  • Spring Data JPA auto-generates SQL based on method names.
  • Only works for READ/DELETE operations.
  • For INSERT/UPDATE, use .save().

üß† Method Name Format:

find | read | get | count | exists | delete | remove
   +
PropertyName (CamelCase)
   +
By <FieldName> [And | Or | Between | In | Like ...]

‚úÖ Examples:

List<UserDetails> findByName(String name);

List<UserDetails> findByNameAndPhone(String name, String phone);

List<UserDetails> findByNameOrPhone(String name, String phone);

List<UserDetails> findByNameIn(List<String> names);

List<UserDetails> findByNameLike(String pattern);

üî∂ 2. üî• JPQL (Java Persistence Query Language)

‚úÖ What is it?

  • Similar to SQL but operates on Entity classes & field names, not DB tables/columns.

‚úÖ Syntax:

@Query("SELECT u FROM UserDetails u WHERE u.name = :name")
List<UserDetails> findByNameJPQL(@Param("name") String name);

‚úÖ Join Example (OneToMany):

@Query("SELECT u FROM User u JOIN u.orders o WHERE o.status = :status")
List<User> findUsersWithOrderStatus(@Param("status") String status);

‚úÖ Custom DTO Projection:

@Query("SELECT new com.dto.UserDTO(u.name, u.phone) FROM User u")
List<UserDTO> getUserDto();

‚úÖ Deletion with JPQL:

@Transactional
@Modifying
@Query("DELETE FROM UserDetails u WHERE u.name = :name")
void deleteByName(@Param("name") String name);

üî∂ 3. üí£ N+1 Problem (One of the most asked!)

‚úÖ Problem:

  • You fetch a list of users (1 query), but for each user, JPA lazily fetches children (N queries).

üß® Total = N + 1 queries

‚û°Ô∏è Bad performance, unnecessary DB hits.

‚ùå Example:

List<User> users = userRepository.findByName("John");
for(User u : users) {
    u.getAddresses().size(); // Each triggers a query
}

‚úÖ Solutions:

1Ô∏è‚É£ JOIN FETCH (JPQL)

@Query("SELECT u FROM User u JOIN FETCH u.addresses WHERE u.name = :name")
List<User> findUsersWithAddresses(@Param("name") String name);

‚úÖ Fetches User + Addresses in 1 single query

2Ô∏è‚É£ @BatchSize

@OneToMany(fetch = FetchType.LAZY)
@BatchSize(size = 10)
private List<Address> addresses;

‚úÖ Groups queries, reduces N+1 to fewer batch-sized chunks.

3Ô∏è‚É£ @EntityGraph

@EntityGraph(attributePaths = {"addresses"})
List<User> findByName(String name);

‚úÖ Declarative way to solve N+1, especially in derived methods.

üî∂ 4. üîÅ @Modifying and @Transactional

‚úÖ When do you need it?

  • For DELETE, UPDATE, or INSERT using JPQL
  • Otherwise, JPA will throw: Query must be a SELECT

‚úÖ Example:

@Transactional
@Modifying
@Query("UPDATE UserDetails u SET u.phone = :phone WHERE u.name = :name")
void updatePhoneByName(@Param("phone") String phone, @Param("name") String name);

üî∂ 6. üöÄ Advanced Joins (Nested Relationships)

@Query("SELECT a FROM A a JOIN a.bList b JOIN b.cList c WHERE c.prop = :value")

‚úÖ JPQL supports multi-level joins like SQL, on mapped entity fields.

üî∂ 9. Top Interview Questions

üí¨ What are derived queries in Spring Data JPA?

Queries automatically generated by Spring from method names based on naming conventions.

üí¨ How do you solve N+1 problem?

Use:
  1. JOIN FETCH in JPQL
  1. @BatchSize
  1. @EntityGraph

üí¨ When do you use @Modifying?

For non-select queries (update/delete/insert) using @Query.

üí¨ Can you paginate and sort derived query results?

Yes, use Pageable and Sort as method parameters.

üí¨ Difference between JPQL and SQL?

JPQL is entity-based and DB-independent. SQL is table-based.
