🔶 1. ✅ Native Queries in JPA

📌 What is it?

  • Plain SQL query that interacts directly with the database.
  • Defined via @Query(nativeQuery = true) or @NamedNativeQuery.

🔎 When to Use Native SQL?

Use Case Reason
Complex DB-specific features JSONB, Window Functions, LATERAL joins, etc.
No Entity Mapping Exists Querying non-entity tables or views
Bulk operations Insert/update/delete multiple records efficiently
Partial data fetch When only few columns needed, not full entity

✅ Simple Native Query Example

@Query(value = "SELECT * FROM user_details WHERE name = ?1", nativeQuery = true)
List<UserDetails> findByNameNative(String name);

🔧 Partial Result Mapping Options:

  1. @SqlResultSetMapping + @NamedNativeQuery (static mapping)
  1. Manual Mapping via:
    • Object[] projection
    • DTO with constructor expression
@Query(value = "SELECT name, phone FROM user_details", nativeQuery = true)
List<Object[]> getPartialData();  // Manually extract fields

✅ Pagination + Sorting in Native Query 

@Query(value = "SELECT * FROM user_details ORDER BY name LIMIT ?1 OFFSET ?2", nativeQuery = true)
List<UserDetails> getPagedUsers(int limit, int offset);

🔶 2. 🧠 Criteria API

📌 Why use Criteria API?

  • For building dynamic, type-safe, DB-independent queries.

🔧 Hierarchy

Component Purpose
CriteriaBuilder Entry point for query construction
CriteriaQuery Represents the query
Root<T> Represents the FROM clause entity
Predicate WHERE clause filtering
TypedQuery<T> Executes the query

✅ Example (Dynamic Filter)

CriteriaBuilder cb = entityManager.getCriteriaBuilder();
CriteriaQuery<UserDetails> cq = cb.createQuery(UserDetails.class);
Root<UserDetails> root = cq.from(UserDetails.class);

// WHERE clause: name = "John"
Predicate condition = cb.equal(root.get("name"), "John");
cq.where(condition);

// Execute
TypedQuery<UserDetails> query = entityManager.createQuery(cq);
List<UserDetails> result = query.getResultList();

🔧 Predicate (Filter) Methods

Operation Criteria API Call SQL Equivalent
Equal cb.equal(root.get("name"), "John") WHERE name = 'John'
Not Equal cb.notEqual(...) WHERE name != 'John'
Greater Than cb.gt(...) WHERE age > 30
Less Than cb.lt(...) WHERE age < 30
Like cb.like(root.get("name"), "J%") WHERE name LIKE 'J%'
In cb.in(root.get("id")).value(1).value(2) WHERE id IN (1, 2)
And cb.and(p1, p2) WHERE cond1 AND cond2
Or cb.or(p1, p2) WHERE cond1 OR cond2

✅ Pagination with Criteria

query.setFirstResult(offset);  // skip first n records
query.setMaxResults(limit);    // limit result count

🔶 3. ✅ Specification API (Clean Layer over Criteria API)

📌 Benefits:

Problem Solved How Specification API Helps
❌ Code duplication ✅ Reuse filters across methods/classes
❌ Boilerplate verbosity ✅ Cleaner abstraction over Criteria API

✅ Setup

  1. Extend your repository:
public interface UserDetailsRepository extends JpaRepository<UserDetails, Long>, JpaSpecificationExecutor<UserDetails> {}
  1. Create Specification:
public class UserSpecifications {
    public static Specification<UserDetails> hasName(String name) {
        return (root, query, cb) -> cb.equal(root.get("name"), name);
    }

    public static Specification<UserDetails> phoneStartsWith(String prefix) {
        return (root, query, cb) -> cb.like(root.get("phone"), prefix + "%");
    }
}
  1. Use in Service:
List<UserDetails> users = repo.findAll(
    where(UserSpecifications.hasName("John"))
        .and(UserSpecifications.phoneStartsWith("123"))
);

✅ Combine Specifications:

Specification<UserDetails> spec = hasName("John").or(phoneStartsWith("9"));

✅ With Pagination:

Page<UserDetails> page = repo.findAll(spec, PageRequest.of(0, 10));

🔶 4. 🔥 Interview Questions Cheat Sheet

💬 When do you use Native SQL instead of JPQL?

When query is DB-specific (e.g. JSONB, recursive joins)When no mapping is availableWhen performance-critical (bulk ops, indexes)

💬 Difference between JPQL and Native SQL?

Feature JPQL Native SQL
DB Independent ✅ ❌
Uses Entities ✅ (field names) ❌ (table & column names)
Custom DTOs ✅ via constructor queries ✅ but requires manual mapping
Complex Joins Limited Full SQL power

💬 What is Criteria API vs Specification API?

Criteria API is low-level, verbose, and gives full control.Specification API is a high-level abstraction over Criteria API with reusability and clean separation.

💬 How do you solve dynamic query filtering?

Use Criteria API or Specification APIAvoid building SQL strings manually (unsafe)

💬 How to fix boilerplate code in Criteria API?

Use Specification APICompose small reusable filters with .and(), .or()
