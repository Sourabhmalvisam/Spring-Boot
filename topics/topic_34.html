Spring Security (Part-4) [Basic Authentication]

🔐 Basic Authentication – Spring Security (Stateless)

✅ 1. What is Basic Authentication?

  • Stateless authentication mechanism.
  • Server does NOT maintain session (no JSESSIONID).
  • Client must send username & password in every request.
  • Format:
    Authorization: Basic base64(username:password)
⚠️ Base64 is not encryption, just encoding — insecure over plain HTTP.

📦 2. HTTP Header Format

Authorization: Basic dXNlcm5hbWU6cGFzc3dvcmQ=
  • dXNlcm5hbWU6cGFzc3dvcmQ= → is base64 of username:password
🚨 Because credentials are sent on every request, HTTPS is mandatory to avoid interception.

🔄 3. Flow of Basic Authentication

🧠 Authentication Flow (Spring Security)

Client sends → Authorization: Basic <base64(username:password)>
↓
BasicAuthenticationFilter
↓
AuthenticationManager (via ProviderManager)
↓
DaoAuthenticationProvider
↓
UserDetailsService (fetch user from DB/memory)
↓
PasswordEncoder (match passwords)
↓
SecurityContextHolder (holds Authentication object)
↓
AuthorizationFilter (checks role/authority)
↓
Controller
⚙️ Everything is handled by Spring Security filters, you write almost no custom code unless you want to override defaults.

🔗 4. Why Credentials are sent in Header (Not Body)?

ReasonExplanation
✅ RFC 7617 StandardWidely supported and standardized
✅ Header not loggedRequest body might be logged by server; headers usually are not
✅ Compatible with all HTTP methodsGET, DELETE, etc. don't support request body

🛠️ 5. Setup – What’s Needed?

➕ Dependencies (pom.xml)

<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-security</artifactId>
</dependency>

⚙️ application.properties

spring.security.user.name=ganesh
spring.security.user.password={noop}pass123

🔁 Or define custom user in DB:

Use UserDetailsService + BCryptPasswordEncoder.

🔍 6. Filter Involved

FilterPurpose

DaoAuthenticationProviderValidates username & password

PasswordEncoderHashes and compares password

🔑 7. Key Classes Involved

  • BasicAuthenticationFilter
  • AuthenticationManager (Interface)
  • ProviderManager (Default impl)
  • DaoAuthenticationProvider
  • UserDetailsService (loadUserByUsername())
  • InMemoryUserDetailsManager / JdbcUserDetailsManager
  • PasswordEncoder (BCryptPasswordEncoder, NoOpPasswordEncoder)

🛡️ 8. Is CSRF Required?

  • ❌ NO – CSRF protection is disabled by default in stateless auth.
  • Because there's no session, there's no attack vector for CSRF.

🚫 9. Disadvantages of Basic Authentication

ProblemExplanation
❌ Credentials in every requestIncreased risk if HTTPS not used
❌ Base64 ≠ encryptionEasy to decode
❌ Stateless = DB hit on every requestAffects performance
❌ No token revocationIf compromised, must change password
❌ Not scalableRepeated password checks increase latency
❌ Size overheadAuthorization header increases request size

✅ 10. When to Use Basic Authentication?

✅ Use it in:

  • Simple internal tools or systems
  • Secured microservices behind gateway
  • Proof-of-concept or test environments

❌ Don’t use in:

  • Public APIs
  • High-traffic or distributed systems
  • Systems needing token revocation or multi-device login

🔄 11. Difference Between Basic and Form-Based Authentication

FeatureBasic AuthForm Login
Session❌ No session (Stateless)✅ Maintains session (Stateful)
Auth FormatHeader – Base64 encoded credsForm POST – username/password
Request FrequencySent on every requestSent once, then session used
Suitable for APIs✅ Yes❌ Not ideal
Custom Login UI❌ No✅ Yes
CSRF Required❌ No✅ Yes (enabled by default)
Scalability❌ Not ideal for large scale❌ Needs session storage

🧠 12. Sample Curl Request

curl -u ganesh:pass123 http://localhost:8080/api/users
  • u automatically adds the base64 encoded header.

🔐 13. Best Practices

✅ Use HTTPS always

✅ Use BCrypt for passwords

✅ Avoid NoOpPasswordEncoder in prod

✅ Avoid basic auth in large scale distributed systems

✅ Switch to JWT/OAuth2 for scalable stateless authentication
