‚úÖ 1. What is JWT Authentication?

  • JWT is a Stateless Authentication mechanism.
  • Server does NOT maintain session state (no JSESSIONID).
  • Client sends JWT token with every request in the Authorization header.

üß± 2. JWT Structure

A JWT has 3 parts:

HEADER.PAYLOAD.SIGNATURE
Part Contains
Header Metadata: algorithm (e.g., HS256, RS256)
Payload Claims: userId, roles, expiry time, etc.
Signature Signed hash to prevent tampering

‚öôÔ∏è 4. Key Components to Implement

‚úÖ Token Generation:

  • Authenticate using UsernamePasswordAuthenticationToken
  • Use AuthenticationManager and DaoAuthenticationProvider
  • On success ‚Üí generate token using JWT utility class

‚úÖ Token Validation:

  • Create Custom Filter to intercept each request
  • Parse JWT from Authorization header
  • Validate:
    • Expiry
    • Signature
    • Claims
  • If valid:
    • Load UserDetails
    • Create Authentication object
    • Set in SecurityContext

üîß 6. AuthenticationManager Logic

  • Delegates to multiple AuthenticationProviders
  • Each Provider must implement supports() method to indicate whether it can handle the request
  • In JWT, we use:
    • Default DaoAuthenticationProvider for username/password login
    • Custom JwtAuthenticationProvider for token validation

üõ°Ô∏è 7. Authorization

  • After authentication, AuthorizationFilter ensures role-based access.
  • Same as form or basic auth.
  • E.g.,
http.authorizeHttpRequests()
  .requestMatchers("/admin/**").hasRole("ADMIN")
  .requestMatchers("/user/**").hasAnyRole("ADMIN", "USER");

üìÇ 9. JWT Implementation ‚Äì What Spring Boot does NOT provide

Spring Security doesn't include JWT by default because:

  • Token structure is app-specific
  • Signing strategy may vary (HMAC vs RSA)
  • Claims (e.g., email, userId) differ
  • Refresh strategies vary per use case
‚ö†Ô∏è You must implement it using custom filters, providers, utilities.

Problem Why

‚ùå Can be hijacked If HTTPS not enforced

üîÅ 12. Refresh Token Strategy

Token Type Duration Stored in
Access Token Short-lived (e.g., 5-15 mins) Header
Refresh Token Long-lived (e.g., 7-30 days) DB or Secure Cookie
Refresh token is used to generate new access token without login.

üß™ 14. Sample Token Generation Code

String token = Jwts.builder()
  .setSubject(user.getUsername())
  .claim("role", user.getRole())
  .setIssuedAt(new Date(System.currentTimeMillis()))
  .setExpiration(new Date(System.currentTimeMillis() + 1000 * 60 * 60)) // 1 hour
  .signWith(SignatureAlgorithm.HS256, SECRET_KEY)
  .compact();
